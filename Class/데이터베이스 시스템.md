# chapter 01 데이터 베이스 환경
## 1.1 정보 처리 시스템
### 정보와 데이터
- 정보와 데이터는 다름
- ```I = P(D)``` , I= 정보 P = 처리기, D = 데이터
- 데이터(D) -> 처리기(P) -> 정보(I)
#### 데이터
- 현실세계로부터 단순한 관찰이나 측정을 통해서 수집된 사실(fact)이나 값(value)
- 숫자, 문자, 스트링, 텍스트, 이미지, 그래픽스
- 오늘날 데이터가 자본, 사람, 기술과 더불어 기업의 4대 요소로 된 것은 기업의 경영에 관한 의사 결정에 중요한 역할을 하기 때문
- 따라서 미국과 일본에서와 같은 첨단 산업국에서는 대부분의 기업들이 별개의 독립 부서로 하여금 전 기관의 데이터를 관리하게 하고 그 책임자를 CIO(Chief Information Officer)라고 하여 최고 경영자급으로 보직
- 이러한 사실들이 데이터의 중요성을 나타냄
#### 정보
- 어떤 상황에 대한 적절한 결정을 할 수 있게 하는 지식으로서 데이터의 유효한 해석이나 데이터상호간의 관계를 의미 
- 데이터를 처리하여 얻은 결과
- 정보 추출 방법을 보통 데이터 처리(data processing) 또는 보다 넓은 의미로 정보 처리(information processing)
- 정보를 통해 유용한 결과(유용성)를 가져오기 위해 데이터의 정확성과 현재성이 필요
### 정보시스템
- 조직에 필요한 데이터를 수집, 조직, 저장하여 필요시에 처리하여 의사결정에 유효한 정보를 생성하여 분배하는 수단
- 조직체의 정보 시스템은 많은 응용 프로그램으로 구성
- 서로 다른 응용프로그램들은 상이한 데이터만 요구하고 처리하는 것만이 아닌, 동일한 데이터를 공용하는 경우도 많음
- ex) 경영 정보 시스템(MIS : Management Information System) : 기업의 경영관리(업무 기획, 운영, 통제)에 필요한 의사 결정을 위해 가장 정확하고 최신의 정보를 제공하는 수단으로 사용되는 정보시스템
- ex) 의사 결정 시스템(DSS: Decision Support System): 전문적인 목적보다는 복합적이고 광범위한 의사 결정 목적을 위해 통합적인 기능을 수행하는 정보 시스템, 광의의 정보시스템
	- 데이터 웨어하우스 : (DSS가 효율적으로 운영되기 위해서) 다양한 소스의 데이터를 별도로 추출하여 관리하는 것
	- 데이터 마이닝 : (데이터 웨어하우스가 대형화되고 복잡해질 때)데이터를 찾고 필요한 정보 또는 지식을 생성하는 과정
- ex) 지식관리 시스템(knowledge management system): 데이터보다 한 단계 수준 높은 지식을 직접 저장하고 관리하여 활용하는 것
#### 응용 시스템(application system)
- 한 부서가 필요로 하는 정보를 제공하여 주는 서브 시스템, 즉 정보 시스템의 일부
#### 응용 프로그램(application program)
- 응용 시스템의 기능을 수행하는 프로그램

## 1.2 데이터 처리 시스템
- 정보시스템은 데이터를 처리하는 서브시스템을 기초로 구성되는데, 데이터 처리는 보통 컴퓨터에 의한 자동 처리를 의미하고, 이것을 초창기에 ```전자 자료 처리 시스템(EDPS : Electronic Data Processing System)```이라고 함
- 데이터가 조직되고 접근되는 방법에 따라 ```일괄처리```, ```온라인 처리```, ```분산 처리 시스템```으로 구분
### 일괄 처리 시스템(batch processing)
- 데이터를 수집, 분류, 정렬한다음 일괄 처리하는 데이터 처리 방법
- 트랜잭션을 한곳에 모아 일정 시간에 한꺼번에 처리함으로써 사용자의 요구를 만족시키고 시스템의 효율성을 높임
- 트랜잭션 : 논리적 작업 단위로서 하나의 단위로 처리되야 하는 분리될 수 없는 연산 그룹
- 순차 접근 방법에 유리 ex) 자기 테이프
- 시스템 측면: 시간 당 처리되는 작업 수가 많아 시스템 성능을 높일 수 있음
- 사용자 측면: 데이터가 즉시 처리되지 않기 때문에 대기시간이 김
- 긴 처리시간, 트랜잭션 작성에 따른 시간 소적인 작업량, 데이터 수록에 대한 경비 등을 보완하기 위해 온라인 시스템이 개발
- 데이터의 입력이 전송을 통해 직접 이루어진다면 온라인 일괄 처리 시스템(on-line batch processing system)라고 함
#### 장점
- 처리비용이 낮음
#### 단점
- 긴 처리시간
- 사전작업 필요
	- 원시 데이터(raw data)를 한곳에 수집하여 분류
	- 분류된 데이터는 컴퓨터가 판독할 수 있도록 기록 정리하여 파일에 수록
	- 이 과정은 순차적이 아니고 병행적으로 수행될 수 있음

### 온라인 처리 시스템(on-line processing system)
- 사전 준비 작업 없이 곧바로 데이터를 처리하는 방식
- 데이터가 생성되면 그 즉시 컴퓨터로 전송되어, 컴퓨터가 즉시 처리해서 그 결과를 원하는 곳으로 보내 주는 시스템
- 처리 결과를 기다렸다가 바로 다음 의사 결정에 반영할 수 있기 때문에 이것을 ```온라인 실시간 처리(on-line real time processing)```라고도 함
- 데이터 처리를 위한 대기시간이 없기 때문에 사용자로서 편리한데, 이런 의미에서 ```사용자 중심 처리 방식```이라고 함
- 계속적인 연구와 시스템 신뢰성 증진으로 온라인 처리 시스템은 ```데이터 처리 방식의 기본```
#### 장점
- 데이터 입력에 따른 데이터의 ```분류```, ```정리작업``` ```절약```
- 데이터 입력이 바로 이루어지기에, 데이터 오류도 입력 시에 즉시 발견, 교정으로 ```회전주기 단축```
- 질의, 검색, 갱신이 가능해 데이터의 현재성을 편리하게 유지할 수 있어 ```효과적인 데이터 처리결과 제공```
#### 단점
- 많은 원격 터미널과 통신 유지 
- ```통신 제어기 필요```
- 짧은 응답시간을 위해 시스템이 계속적으로 가동 상태를 유지, 잠시의 정지도 허용x -> ```작업당 처리 비용 증가```
- 모든 단말 장치의 통신이 항상 가동 상태로 유지 해야함 -> ```프로그램의 테스트나 보수 유지 어려움```
- 이런 문제들이 시스템 오버헤드로 연결되어 ```시스템 성능 저하```

### 분산 처리 시스템
#### 중앙 집중 시스템(centralized system)
- 전통적인 데이터 처리시스템
- 데이터가 저장된 곳을 중심으로 기능들이 집중되는 것
- 하나의 서버에 여러 개의 클라이언트가 공용하는 형태
##### 장점
- 작업 처리 결과의 범조직적 통합
- 데이터 처리 업무 통제의 용이
- 인력 효율적 이용
- 통신비용 절약
##### 단점
- 데이터가 지리적으로 분산되어 있는 환경에서는 효과적x
#### 분산 처리 시스템(distributed processing system)
- 지리적(물리적)으로 분산되어 있는 ```처리기와 데이터베이스를 네트워크로 연결```시켜 사용자가 ```하나의 시스템을 사용하는 것처럼``` 데이터를 처리해 주는 시스템
- 분산처리기, 통신 네트워크, 분산 데이터베이스로 구성
- 각 처리기는 데이터의 처리 요구에 따라 클라이언트도 되고 서버도됨
- 분산 처리시스템을 클라이언트/서버 시스템으로 표현
- 각 시스템(노드)이 완전히 별개로 독립, 분할 통제되는 것이 아니라,  하나의 구성원으로서 범 시스템적인 규정에 따라 ```통제```
- ex) 은행, 항공 회사
##### 장점
- 지역 문제 발생에 대한 신속한 조치
- 지역 책임 구분
- 새로운 응용에 대한 모듈식 구축의 용이
- 노드의 폐쇄, 확장등이 시스템 영향 최소화
- 지속적으로 확장 발전 가능

##### 분산 처리기(distributed processor)
- 지리적으로 분산 설치되어 있는 복수의 컴퓨터, 즉 ```지역 처리기(local processor)```를 가리킴
- 그 지역의 데이터를 처리하는 지역 컴퓨터로서 하나의 ```독립된 데이터 처리 시스템의 중추 역할```
##### 통신 네트워크(communication network)
- ```분산 처리기들을 연결``` 시켜 ```자원```을 ```공유```하여 논리적으로 ```하나의 시스템```과 같이 ```운영```되도록 하는 네트워크
- 이를 구성하는 각 노드는 독립된 처리기 능력을 가진 지역 컴퓨터 시스템이기 때문에, 통신 네트워크는 ```컴퓨터 네트워크```를 말함
- 통신 네트워크에 연결되어 있지 않고 분산되어 있는 시스템을 비 중앙 집중 시스템(decentralized system)
- 통신 네트워크의 모든 지역 처리기들은 특정 ```통신 규약```에 따라 데이터를 전송, 수신 하도록 되어있음
- 통신망의 구조는 데이터 통신에 상당한 영향을 미침
##### 분산 데이터베이스(distributed database)
- 데이터가 지리적으로 분산 저장되어있는 형태의 데이터베이스
##### 서버
- 데이터를 ```처리```해 주거나 전송해 주는 컴퓨터 처리기
##### 클라이언트
- 데이터의 처리나 전송을 ```요청```하는 컴퓨터 처리기

## 1.3 데이터베이스의 정의
### 역사
- 1963년 6월 미국SDC회사가 개최한 제 1차 "컴퓨터 중심의 데이터베이스  개발과 관리"(~Data base)라는 제목에서 공식적으로 처음 사용
- 데이터베이스 시스템 용어도 1965년 9월 SDC 제 2차"컴퓨터 중심 데이터베이스 시스템"에서  처음 사용
- 데이터베이스 관리 시스템에 개척자적 업적을 남긴 사람은 G.E(General Electric) 회사의 바크만(Charled W. Bachman)
- 바크만은 1963년 IDS(integrated Data Store)라는 데이터베이스 관리 시스템을 최초로 만들어 현대적 의미의 데이터베이스 환경을 확립
### 정의 
- 데이터베이스는 원래 같은 데이터가 상이한 목적을 가진 여러 응용에 중복되어 사용될 수 있다는 ```공용```의 개념에 기초를 둠
- 한 조직의 여러 응용 시스템들이 ```공용```할 수 있도록 ```통합```, ```저장```된 운영 데이터의 집합  
###### 1. 통합 데이터
- 똑같은 데이터가 원칙적으로 중복되어 있지 않는 것
- ```효율성```을 증진시키기 위해 ```일부 데이터의 중복을 허용```
- 이러한 의도적 중복은 항상 파악, 관리할 수 있고, 이를 ```최소의 중복``` 또는 ```통제된 중복```이라 함
- 데이터 베이스는 통제된 중복이 허용된 통합된 데이터  
###### 2. 저장 데이터
- 컴퓨터가 접근 할 수 있는 저장 매체에 저장된 데이터   
###### 3. 운영 데이터
- 조직은 고유의 ```기능```을 가지고 있고, 그것을 ```수행```하기 위해 반드시 ```유지해야될 데이터```를 의미
- 데이터 베이스는 단순한 데이터의 집합이 아니라, 조직의 존재 목적이나 기능을 수행하는데 없어서는 안 될 데이터의 집합  
###### 4. 공용 데이터
- 조직에 있는 여러 응용 시스템들이 ```공동```으로 ```소유```하고 ```유지```하며, ```이용```하는 공용 데이터
- 사용자들이 서로 다른 목적으로 데이터베이스의 데이터를 ```공동```으로 ```이용```  
## 1.4 데이터베이스의 특성
###### 1. 실시간 접근성(real-time accessibility)
- 임의적이고 비정형적인 질의(query)에 대해 ```실시간 처리```로 응답할 수 있어야함
- 데이터 베이스는 실시간 처리를 위해 활용되는 성질을 가짐  
###### 2. 계속적인 변화(continuous evolution)
- 어느 한 시점에 데이터베이스가 저장하고 있는 내용은 그 데이터베이스의 한 ```상태```를 나타냄 이 데이터베이스의 상태는 ```동적```임
- 데이터베이스는 데이터 삽입, 삭제, 갱신으로 항상 데이터들이 변하고, ```변화 속에서 현재의 정확한 데이터를 유지해야함```
- 데이터베이스는 항상 변하고 있는 현실 세계를 표현해야 되기에 자연히 계속적으로 변하지 않으면 안됨  
###### 3. 동시 공용(concurrent sharing)
- 사용자들이 원하는 데이터 또는 같은 데이터에  동시(concurrent)에 공용할 수 있어야함
- 데이터베이스의 데이터 요소들이 처음부터 항상 동시공용이 가능하도록 조직, 저장, 관리되어야 함  
###### 4. 내용에 의한 참조(contents reference)
- 데이터베이스의 데이터 참조는 데이터 레코드들의 주소나 위치가 아니라 사용자가 요구하는 ```데이터의 내용``` 또는 데이터가 가진 ```값```에 따라 참조
- 레코드들은 주소나 위치에 관계없이 모두 하나의 논리적 단위 속에 포함되어 접근 검색됨

## 1.5 데이터베이스의 개념적 구성요소
### 개체(entity)
- 개체(entity)는 데이터베이스에서 표현하려고하는 유,무형의 ```객체(object)```로서 ```서로 구별되는 것```을 의미
- 개체는 현실 세계에 대해 사람이 생각하는 ```개념```이나 ```정보의 단위```로서 의미를 가짐
- 컴퓨터가 취급하는 파일의 ```레코드(record)```에 대응됨
- 개체는 단독으로 존재할 수 있으며, 정보로서 역할을 함
- 하나의 개체는 하나 이상의 ```속성(attribute)```으로 구성되고 각 속성은 개체의 특성이나 상태를 나타냄
-ex) 개체: 학생, 속성: 학번, 이름, 학과
- 속성은 ```이름을 가진``` 데이터의 ```가장 작은 논리적 단위```
- 컴퓨터 파일 구조에서는 속성을 데이터 ```항목(data item)``` 또는 ```필드(field)```라고 함
- 속성은 그 자체만으로 중요한 의미를 표현하지 못하기 때문에 단독으로 존재x
- 속성은 시간에 따라 변할 수 있음
- 특정 개체는 개체를 구성하는 속성들이 어떤 구체적인 값을 가짐으로써 실체화됨
- "학생" 개체를 나타내는 <학번:1234, 이름: 홍길동, 학과: 컴퓨터>는 학생 개체의 한 값으로 볼 수 있는데 이를 ```개체 인스턴스(entity instance) 또는 ```개체 어커런스(entity occurrence)```라고 하고 이 인스턴트들의 집합을 ```개체 집합(entity set)```이라 함
- 데이터 베이스는 바로 이런 ```개체 인스턴스```를 저장하는 것
- '학번, '이름', '학과'와 같은 속성 이름들로만 기술된 개체의 정의를 ```개체 타입(entity type)```이라 함
- ```개체 타입```은 ```논리적인 개념```이고 ```개체 인스턴스``는 데이터 베이스에 저장되는 ```구체적인 객체```

### 관계
- 개체 관계와 속성 관계가 존재
- ```개체 관계```는 개체 집합과 개체 집합 사이의 관계이며, 데이터베이스에 저장할 대상
- 개체 관계는 개체들을 서로 연관시켜 ```어떤 의미```를 나타내어 개체와 다름이 없기 때문
- 개체 관계는 보통 무형적이고 다루기 복잡합
- ```속성관계```는 개체를 기술하는 속성들 간의 관계
- 개체 관계는 개체 간 관계(inter-entity relationship)
- 속성 관계는 개체 내 관계(intra-entity relationship)
- 데이터베이스는 보통 ```개체 관계```만 ```명시적```으로 취급, ```속성관계```는 ```암시적```으로 취급
- 개체 관계를 도식으로 표현한 다이어 그램을 E-R 다이어 그램(entity-relationship diagram)이라 함
- 개체 관계는 데이터베이스를 접근할 때 사용하는 중요한 도구
- 개념적으로 ```데이터베이스```는 ```개체```와 ```관계```라는 요소로 구성
- 물리적으로 저장된 ```데이터베이스```에는 ```개체와 관계의 인스턴스``` 또는 어커런스들의 ```집합```들이 저장되어 있음
- ```개체 타입```은 논리적으로 ```무한한 어커런스```를 가지지만 실제로 데이터베이스에 저장되는 것은 ```정보 추출의 대상```이 되는 것으로 ```제한```

## 1.6 데이터베이스의 구조
- 사용자의 입장, 시스템(저장 장치)의 입장에 따라 ```논리적 구조```와 ```물리적 구조```로 구분됨
- 하나의 논리적 구조에 대해 이것을 구현하는 물리적 구조가 아주 다를 수 있고, 여러가지 방법이 있을 수 있음
- 하나의 데이터 베이스를 표현하는 ```논리적 구조```와 ```물리적 구조```는 서로 ```대응관계```를 가져야함
- 그로인해 ```동등성```을 유지
- 이 두 구조는 데이터베이스를 사용하는 사용자와 저장 관리하는 시스템간의 ```통신 수단```이 됨
### 논리적 구조(logical structure)
- 사용자가 생각하는 데이터의 논리적 표현
- ```사용자 입장```에서 보는 데이터베이스 구조로서 데이터가 배치되어 있다고 간주하는 ```가상적인 구조```를 말함
- 이 구조에서 취급하는 데이터 레코드를 ```논리적 레코드(logical record)```
- 논리적 구조로 표현한 데이터베이스를 ```논리적 데이터베이스```라고 함
### 물리적 구조(physical structure)
- 시스템(저장 장치)입장에서 본 데이터베이스 구조로서 저장 데이터의 ```물리적 배치를 표현```한 것
- 디스크나 테이프와 같은 저장 장치 위에 물리적으로 저장되어 있는 ```데이터의 실제구조```
- 이 구조에서 취급하는 데이터 레코드를 ```저장 레코드(stored record)```
- 물리적 구조로 표현한 데이터베이스를 ```물리적 데이터베이스```라고함

# chapter 02 데이터베이스 관리 시스템

## 2-1 데이터베이스 관리 시스템의 발전 배경
### 파일 시스템(file system)
- 초기의 데이터 처리 시스템은 각 응용 프로그램이 개별적으로 자기 자신의 데이터 파일(file)을 ```관리 유지```
- 응용 프로그램은 데이터 파일에 접근하고 관리하기 위한 코드 또한 포함하고 있어야 함
- ```데이터 공용을 할 수 없는``` 파일 시스템의 ```가장 큰 문제점```은 ```데이터 종속성```과 ```데이터 중복성```으로 집약시킬 수 있음
#### 특성
- 응용 프로그램은 논리적 파일 구조를 직접 물리적 파일 구조로 구현해야 함
- 프로그래머는 물리적 데이터 구조에 대해 잘 알고 있어야 데이터 접근 방법을 응용 프로그램에서 효율적으로 구현할 수 있음
- 하나의 파일은 하나의 응용프로그램을 위해 존재, ```즉 1대1 대응```
#### 문제점
##### 데이터 종속성(data depenency)
- 응용 프로그램과 데이터 간의 ```상호 의존 관계```를 말함
- **데이터의 구성 방법이나 접근 방법을 변경**할 때는 **응용 프로그램도 같이 변경**시켜야함
##### 데이터 중복성(data redundancy)
- 한 시스템 내에 **내용이 같은 데이터가 중복되게 저장 관리되는 것**을 데이터 중복성이라 함
##### 파일 관리 시스템(file management system)
- 응용 프로그램이 사용하는 데이터 파일 처리 루틴들을 **하나의 공동 접근 루틴**으로 구성
- 같은 데이터에 대해 응용 프로그램들이 요구하는 상이한 형태의 데이터 구조를 지원하지 못함
- 파일 시스템처럼 **데이터 중복성이 존재**
##### 데이터 중복성의 문제점
##### 1. 일관성(consistency)
- 데이터 중복이 존재하면 **동일성을 유지하기 어려움**
- 결과적으로 ```데이터 간에 불일치```가 일어나 내부적으로 데이터 일관성이 없어짐
- 이러한 데이터 베이스는 ```모순성(inconsistency)```을 갖게됨
##### 2. 보안성(security)
- 한 시스템 내에 존재하는 ```같은 내용의 데이터```에 대해서는 ```똑같은 수준의 데이터 보안이 유지```되어야 함
- 같은 데이터가 여러 곳에 **중복**되어 저장되어 있는 경우 ```모두 똑같은 수준의 보안을 유지하는 것은 매우 어려움```
##### 3. 경제성(economics)
- 데이터를 ```중복 저장```하게 되면 ```추가적으로 필요한 저장 공간```에 대한 ```비용이 더 듦``
- 또한 중복되어 있는 상황에서의 ```갱신 작업```은 ```중복```된 모든 ```데이터```를 ```모두 찾아 전부 수행```해야만 되므로 ```갱신 비용이 높아```지며 ```경제성은 떨어짐```
##### 4. 무결성(integrity)
- 데이터가 중복 저장되면 제어가 분산되어 ```데이터의 무결성```, 즉 ```데이터의 정확성을 유지하기 어려움```
##### 5. 공용성
- 하나의 파일을 ```개방(open)```하여 사용하면 그 프로그램의 실행이 모두 끝나 다시 **폐쇄(close) 할 때 까지 다른 프로그램은 이 파일에 접근 불가**
- 즉 ```데이터의 동시공용을 지원하지 못함```
이러한  파일 시스템의 단점을 보완하고 사용자 입장에서 보다 편리하게 데이터 처리를 수행할 수 있는 새로운 방법이 요구
-------------------
## 2.2 데이터베이스 관리 시스템의 정의
### 데이터베이스 관리 시스템(DBMS : DataBase Management System)
- 파일 시스템에서 야기된 **데이터 종속성과 중복성의 문제를 해결**하기 위해 고안된 시스템
- 응용 프로그램과 데이터의 ```중재자```로서 모든 응용 프로그램들이 데이터베이스를 ```공용```할 수 있게끔 관리해 주는 **소프트웨어 시스템(software system)**
- 응용 프로그램들이 **데이터베이스를 이용하기 위해서는 DBMS를 통해서만 가능**
- 즉, 데이터베이스의 구성, 접근 방법, 관리 유지에 대한 **모든 책임을 DBMS가 지고 있음을 의미**
- 응용 프로그램은 원하는 **데이터와 처리작업만을 명세해서 DBMS에 요청만** 하면 됨
## 2.3 데이터베이스 관리 시스템의 필수 기능
- 데이터베이스를 여러 응용 프로그램들이 ```올바르게 공용할 수 있도록``` 관리하고 지원해 주기 위해서 DBMS는 기본적으로 ```다음 기능을 필수적으로 구비```해야 함
### (1) 정의 기능
- **데이터 정의(data definition) 기능**은 다양한 ```응용 프로그램```과 ```데이터베이스```가 서로 ```인터페이스를 할 수 있는 방법을 제공하는 것```
- 구현된 하나의 물리적 구조의 데이터 베이스로 사용자들의 다양한 데이터 요구를 지원해 줄 수 있도록 ```가장 적절한 데이터베이스 구조를 정의할 수 있는 기능```을 말함
- 응용 프로그램은 이 데이터 정의를 기초로 원하는 데이터를 DBMS에 표현
데이터 정의 특성
1. 모든 응용 프로그램들이 요구하는 데이터 구조를 지원할 수 있도록, ```데이터베이스의 논리적 구조와 특성이 DBMS가 지원하는 데이터 모델(data model)에 맞게 기술되어야 함```
2. 물리적 저장 장치에 저장하기 위한 ```데이터의 물리적 구조와 명세도 포함```해야함
3. 데이터의 ```논리적 구조와 물리적 구조``` 사이에 상호 변환이 가능하도록 두 구조 사이의 ```사상(mapping)을 명세``` 해야함
-  하나의 물리적 구조로 여러 응용 프로그램이 요구하는 데이터 구조를 지원할 수 있는 방법이 사상임
### (2) 조작 기능
- **데이터 조작(data manipulation) 기능**은 ```사용자```와 ```데이터베이스``` 사이의 ```인터페이스를 위한 수단을 제공```
- DBMS는 데이터의 검색, 갱신, 삽입, 삭제 등 ```데이터베이스 연산을 지원하는 도구```로서 **데이터 언어(data language)** 를 통해서 표현될 수 있음
#### 데이터 언어의 요건
1. 모든 사용자가 ```사용하기 쉽고 자연스러워야함```
- 데이터 공용이라는 본래의 목적을 달성하기 위해 사용절차가 쉽고 자연스워러야함
2. ```명확```하고 ```완전```해야함
- 데이터 간의 관계를 명확하게 명세해야하며, 사용자가 원하는 데이터 연산은 무엇이든지 합법적인 범위 내에서 완전하게 명세 해야함
3. ```효율적```이여야함
- 공용된다는 이유로 데이터 접근이나 처리가 비효율적으로 수행되면 안됨
### (3) 제어 기능
- DBMS는 ```공용 목적```으로 관리되는 데이터베이스 내용에 대해 항상 ```정확성```과 ```안정성```을 **유지**할 수 있는 ```데이터 제어(data control)``` 기능을 가지고 있어야 함
데이터 제어 요건
1. 데이터베이스를 접근하는 갱신, 삽입, 삭제 작업이``` 정확하게 수행```되게 하여 ```데이터의 무결성(integrity)```이 파괴되지 않도록 제어해야함
2. 정당한 사용자가 ```허가된 데이터만```을 ```접근``` 하도록 ```권한(authority)```을 **검사**하고 ```보안(security)```을 **유지**해야함
3. 데이터베이스에 ```동시에 접근```해 데이터를 처리해도 ```데이터베이스와 처리결과```는 ```항상 정확성(correctness)```이 유지되도록 ```병행 제어(concurrency) 기능```을 가져야함
## 2.4 데이터베이스 관리 시스템의 장단점
### 장점
##### 1. 데이터 중복의 최소화
- 데이터를 통합해서 구성함으로써 중복을 사전에 통제 
- 즉 이것은 중복을 완전히 배제한다는 것이 아닌, 성능 향상의 이유로 데이터 중복이 불가피할 때가 있음
- 시스템은 어떤 데이터가 어느 곳에 중복되어 있는지 알고 있기 때문에 이와 같은 중복을 제어된 중복(controlled redundancy)라고 함
##### 2. 데이터의 공용
- 같은 내용의 데이터를 다양한 구조로 지원할 수 있는 DBMS의 기능덕분에 데이터베이스의 ```데이터 공용(data sharing)이 가능```
- 데이터의 공용으로 응용 프로그램들이 수행하던 데이터에 대한 ```유지관리 부담을 면제```, **새로운 응용 프로그램**에 대해서도 데이터 구성에 신경 쓸 필요 없이 ```응용 자체에만 전념```할 수 있도록 도와줌
##### 3. 데이터의 일관성 유지
- 데이터의 ```중복을 제어```하고 ```통제```를 통해 ```데이터의 일관성(data consistency) 유지```
##### 4. 데이터 무결성 유지
- ```데이터 무결성(data integrity)```은 ```저장된 데이터 값```과 ```현실 세계의 실제 값```이 일치하는 ```정확성(accuracy)```을 말함
- ```데이터의 중복성```이 **완전히 제거**되도 ```허용되지 않은 값```이나 ```부정확한 데이터```가 여러가지 경로에 의해 데이터베이스에 잠입될 수 있음
- ```DBMS```는 데이터베이스가 ```접근 처리```될 때마다 ```제어 기능```을 통해 ```유효성```을 ```검사```함으로써 ```데이터 무결성을 유지```
##### 5. 데이터의 보안 보장
- DBMS는 데이터베이스를 ```중앙 집중식```으로 ```총괄```, ```관장```함으로써 ```데이터베이스의 관리 및 접근```을 ```효율적으로 제어```할 수 있음
- DBMS가 ```정당한 사용자```, ```허용된 데이터```와 ```연산``` 등을 ```확인 검사```함으로써 ```모든 데이터```에 대해 ```철저한 보안 유지```를 보장함
##### 6. 표준화
- DBMS의 ```데이터 제어 기능```을 통해 데이터(데이터 형식, 내용, 처리 방식, 문서화 양식 등)에 관한 ```표준화를 범 기관적으로 쉽게 시행```
##### 7. 전체 데이터 요구의 조정
- 기관의 모든 응용 시스템들이 요구하는 ```데이터```들을 ```기관 전체에 가장 유익한 구조```로 ```조직```하여 ```효율적인 정보 처리 효과```를 얻게 할 수 있음
### 단점
##### 1. 운영비의 증대
- DBMS는 ```고가의 소프트웨어 제품```이며, 컴퓨터 시스템의 ```자원을 많이 사용함```
- 특히 메인 메모리를 많이 차지해서, DBMS를 운영하기 위해 ```메모리 용량이 많이 필요```하고, 더 ```빠른 CPU를 요구```함
- 결과적으로 ```시스템 운영비의 오버헤드를 가중```시킴
##### 2. 특정 응용 프로그램의 복잡화
- 데이터 베이스는 상이한 여러 유형의 데이터가 서로 관련되어 있음
- 이런 상황 속에서 ```특정 응용 프로그램```은 ```여러 제한점```을 가지고 ```작성```되고 ```수행```될 수 있음
- 따라서 ```특수 목적의 응용 시스템```은 설계 기간이 길어지고 보다 ```전문적```, ```기술적```이 되어야 하므로 그 ```구조가 복잡```하게 되고 ```성능이 저하```될 수 있음
##### 3. 복잡한 백업과 회복
- 데이터베이스는 ```구조가 복잡```하고 여러 사용자가 ```동시에 공용```하기 때문에 ```에러```가 발생했을 때 **정확한 원인과 상태를 파악하기 어려움**
- 이에 대한 ```백업(backup)조치와 사후 회복(recovery) 기법```을 **정교하게 수립하지 않으면** DB 이용에 `**막대한 지장을 초래**할 수 있음
##### 4. 시스템의 취약성
- 데이터베이스 시스템은 ```통합된 시스템```이기 때문에 ```일부에 에러```가 발생할 경우 ```전체 시스템이 정지``` 또는 ```영향```을 받아 ```시스템의 신뢰성과 가용성을 저해```할 수 있음 
- 데이터베이스에 의존도 높은 환경에서 치명적인 약점임
요약
- 특수한 성능이나  데이터 처리를 요건으로 하는 응용 시스템에서는 DBMS를 채택하지 않을 수 있음
- 그러나 현재의 **대부분 대규모 데이터 관리와 처리가 요구**되는 상황에서는 **DBMS가 불가피한 도구**가 되었으며, 이러한 **문제점에 대한 대처가 잘 준비**되어 있다고 평가받고 있음
## 2.5 데이터 독립성
- DBMS가 추구하는 **궁극적인 목적**은 **데이터의 논리적 구조나 물리적 구조가 변경되어도 응용 프로그램이 영향을 받지않는 (또는 영향을 최소화하는)** ```데이터 독립성```을 제공하는 것
- 데이터 독립성은 논리적 데이터 독립성과 물리적 데이터 독립성으로 나우어 생각할 수 있음
### 논리적 데이터 독립성(logical data independency)
- 공용하고 있는 데이터베이스의 논리적 구조를 변경해야될 때, 기존 모든 응용 프로그램의 논리적구조를 변경하거나 응용 프로그램을 재 작성해야된다면 어려움에 문제가 생김
- **논리적 데이터 독립성**이란 DBMS가 데이터베이스의 논리적 구조를 변경시키더라도 **기존 응용 프로그램들에** ```영향을 최소화(영향이 완전 없진 않음)```하는 것을 말함
- 논리적 데이터 독립성은 **하나의 논리적 데이터 구조를 가지고**, 여러 응용 프로그램이 요구하는 **다양한 형태의 논리적 구조로** ```사상```시켜 지원해 줄 수 있는 능력이 있을 때 가능
### 물리적 데이터 독립성(physical data independency)
- ```하나의 논리적 구조```로 표현된 데이터베이스는 ```하나의 물리적 구조```로 구현되어야함
- 데이터베이스는 ```하나의 구현된 물리적 구조```로 **여러 응용 프로그램들을 지원해야만 함**
- 그러나 **새로운 저장 장치의 개발이나 접근 방법의 개발**로 **성능을 개선시키기 위해 데이터의 물리적 구조를 변경해야 될 때가 있음**
- ```물리적 데이터 독립성```이란 **기존 응용 프로그램들에** ```아무런 영향을 주지 않고``` DBMS가 데이터베이스의 **물리적 구조를 변경할 수 있는것을 말함**
- 또한 물리적구조의 변경이 논리적 구조에 영향을 주어서는 안됨
- 즉 ```물리적 데이터 독립성```은 ```응용프로그램```이나 ```DB의 논리적 구조```에 ```영향을 주지 않고``` DBMS가 **데이터 물리적 구조를** ```변경```할 수 있는것을 말함
- 이것은 ```하나의 논리적 구조```와 이를 지원할 수 있는 여러가지 상이한 **물리적 구조 사이의** ```사상(mapping)능력```이 있어야 가능
## 2.6 데이터베이스 관리 시스템의 역사
### 1960년대 초반 GE(General Electric)
- 최초의 범용 DBMS는 1960년대 초 GE의 버크만에 의해 설계된 ```IDS(Integrated Data Store)```임
- 이것은 데이터 표현의 표준 모델인 ```네트워크 데이터 모델(network data model)```의 ```기초```가 됨
- 또한 버크만은 데이터 모델을 정확하게 명세할 수 있는 방법으로 유명한 데이터 구조도(data structure diagram)를 제안함
### 1960년대 후반 IBM
- 1960년대 후반에 IBM은 ```IMS(Information Management System)``` DBMS을 개발하였고, 이것은 **현재에도 상당히 많은 대형 응용 시스템에서 이용되고 있음**
- IMS는 ```계층 데이터 모델(hierarchical data model)```이라고 하는 또 다른 데이터 표현 기법의 ```기초```가 됨
### 1970년대 제1세대 DBMS & 관계형 데이터 모델
- 대형 컴퓨터 제조 회사들이 DBMS를 제작 판매
- ```네트워크 데이터 모델```과 ```계층 데이터 모델```에 **기초**를 둔 DBMS를 ```제1세대 DBMS(first-generation DBMS)```함
- ```IBM San Jose 연구소```의 ```E. F. Codd```는 ```관계 데이터 모델(relational data model)```이라는 **새로운 데이터 표현 방법**을 제안
- 이 ```관계 데이터 모델```은 ```데이터베이스의 기초```과 되었고, ```DBMS 개발의 전환점```이 됨
- **관계데이터베이스의 이점**이 인식되면서 한 조직의 데이터 관리에 **DBMS의 사용이 표준으로 됨**
### 1980년대 제 2세대 DBMS
- **관계 데이터 모델**이 **모든 DBMS의 주류가 됨**
- ```IBM```이 ```관계DBMS```의 일부로 개발한 ```SQL(Structured Query Language)```은 ```세계 표준 데이터베이스의 언어```가 됨
- 대부분의 DBMS는 하드웨어에 독립적으로 운영되도록 개발됨
- 80년대에는 ```PC를 기반```으로 하는 DBMS가 출현했는데, 이 관계 데이터 모델을 기반으로 한 DBMS를 ```제 2세대 DBMS```라고 함
### 1980년 후반 ~ 1990년대 제 3세대 DBMS
- **이미지, 텍스트, 그래픽스**와 같은 ```새로운 데이터 타입```을 **저장**하고 ```복잡한 질의```를 **처리**할 수 있도록 ```기존의 DBMS 기능들이 확장```
- 데이터베이스 응용에 대한 ```복잡성```이 **계속 증대**되고, ```사용자의 요구```에 **대처**하기 위해 ```객체 데이터 베이스(ODBMS: Object Database Management System)```와 곧이어 ```객체-관계 데이터베이스(ORDBMS: Object-Relational Database Management System)```이 출현
- 이렇게 새로 개발된 DBMS를 제3세대 DBMS(third generation DBMS)라고 함
- 현재는 제2세대 DBMS와 제3세대 DBMS를 공존하여 사용
